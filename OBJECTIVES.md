# Objectives

> **Back-of-mind goals that every GSD session should be aware of.** These aren't tasks — they're the destination. Every decision, every piece of work should be pulling toward these.

## The Vision

Token Remote is becoming a **multi-machine AI development platform**. One human, many machines, many GSD agents, all coordinated. The system gets smarter over time.

## End Goals

### 1. Multi-Machine CLI Commander
Send tasks to any connected machine from the mobile app. One human, many machines, all coordinated. Hivemind is the shared brain. Token Remote is the nervous system.

### 2. Agent Workflow Engine
Define processing flows (agent chains with pass/loop gates). First flow: CLI output → categorize → summarize → analyze → rules. Then planning, code writing, context clearing, build uploads.

### 3. OTA-First Mobile App
Native builds only for native module changes. All UI/logic ships via `eas update`. Settings screen is the control panel.

### 4. Desktop Daemon Supervisor
Menu bar app keeps the daemon alive, auto-restarts on crash, adopts external instances. Download link works from a browser address bar.

### 5. Self-Improving System
The improvement pipeline ingests raw CLI data, refines it through agent flows, and produces rules that make the system better over time. Humans intervene less as the rules mature.

### 6. Feature-Complete Messaging
Clean compose modal with expandable recipient sections (Agents, CLIs, People). Tasks dispatched from messages. No dead Token Sports artifacts.

### 7. Universal Base Models
TokenBase is the shared foundation. All Token projects consume it. Models are the spec for a future compiled language.

## Self-Improvement Mandate

**Every GSD session should be learning.** Not just completing tasks — observing what works and what doesn't.

Track in `knowledge/effectiveness.md`:
- **What worked well** — patterns, tools, approaches that were effective
- **What didn't work** — dead ends, wasted cycles, bad assumptions
- **Model limitations** — things the base model (Claude, etc.) consistently gets wrong or struggles with
- **Process friction** — where GSD's own workflow creates unnecessary overhead
- **Time sinks** — tasks that took way longer than they should have and why

This isn't busywork. The self-improving system (Goal #5) starts here. The observations GSDs collect become the training data for better rules, better prompts, and eventually better agent workflows.
